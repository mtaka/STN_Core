### 1. STNとは
#### 1.1 STNとは

Structured Text Notation（STN）は、テキストで構造を表現するための記法です。
ちょっとした設定データであれ、まとまった文章であれ、同じデータを様々な形で使用したいケースは数多くありますが、テキスト形式は最も基本的な形式であり、アプリケーションによらず利用できるという利点があります。
一方、元来線形なため、そのままでは構造を持つことができません。
これを解決する記法として、人間が読み書きもでき、LLMにも理解されやすく、また決定論的なプログラムのデータとしても使用できるようにしたいと思って定義しています。

従来、構造化テキストのために、様々なフォーマットが提案されて使用されています。
XML、JSON、YAML、Markdownなどです。
もう既に有効に活用されているこれらの形式を置き換えるものではありませんが、これらを補完していけるような記法を目指すものです。


#### 1.2 STNの考え方
STNは、以下のような考え方に基づいて作られています。
- 人間にも、LLMにも、プログラムにも理解できる形式であること
- 階層を持った情報を記述できること。
- 記述した内容が再利用できるよう、変数を持つことができること。
- 特定の形式を守れるよう、型やスキーマを持つことができること。
- 但し、変数や型・スキーマが未定義でも壊れないこと（Success Safe）
- 一度記述した内容を更新せず、注釈的に情報を付加していくことができること。

#### 2. STNの構造
#### 2.1. S式ライクな「単語のリスト」
STNの基本構造は、括弧と空白で区切られた単語の階層的なリストです。
```
(a b c) 
```
これは、多くの言語で `['a', 'b', 'c']` と表される内容と同じ意味になります。
このとき、
- 要素はデフォルトで文字列
- 空白で区切る
という形式にしているため、引用符もカンマも不要です。

### 2.2. 括弧のネスト

括弧はネストできます。
```
(a b c ( d e f ))
```
これは、 `['a', 'b', 'c', ['d', 'e', 'f']] ` の意味になります。

### 2.3. チャンク
リストは、途中`;`を用いることで、チャンクに分割できます。
(a b c ; d e f )
これは、 `[['a', 'b', 'c'], ['d', 'e', 'f']] 

### 2.4. 名前付き要素
ここまでは、配列の入れ子でしかありませんが、要素に名前をつけることができます。

```
(:name Joe :age 36 :sex Male)
```

これは `{'name': 'Joe', 'age': '36', 'sex': 'Maile'}` という意味になります。
要素に名前をつける場合は、名前の先頭に`:`を付加することで名前を明示します。

ここで、名前の対応は名前の次に来る要素との1対1対応となります。
つまり、暗黙的に次の要素とのペアによるチャンクが作られ、名前との対応が取られると考えます。

### 2.5. リテラル
空白を含んだ一連の文字列を表したい場合はリテラルを使います。
STNでは、リテラルに引用符を使わず、[]で囲んで表現します

```
[This is literal string.]
```

引用符では始めと終わりが同じ記号になりますが、[]を使うことで明示的に始めと終わりが明確になります。

#### 2.5.1 リテラル内のエスケープ
リテラル自体はネストしません。
リテラルは `]` で終了するため、この文字が含まれる場合は `\` でエスケープします。

```
[[x\] This is literal string.]
```

は、他の言語だと

```
'[x] This is literal string.'
```

で表されることになります。

#### 2.5.2 リテラルの別表現
リテラル終了マークである`]`が含まれる文字列を表現する手段として、```（バッククオート）によるリテラル表現が用意されます。
```
`[x] This is literal string.`
```
バッククオートとバッククオートで囲まれたテキストがリテラルと見なされます。[]もエスケープせずに含むことができます。
なお、バッククオート自体が含まれる場合はエスケープが必要です。

`[]` によるリテラル表現と異なり、始めと終わりに同じ文字を使用するため、代替手段としての仕様となります。


#### 2.5.3 ブロックリテラル
他のデータソースから貼り付けるなど、の場合、]を含んでいるような場合にそのまま貼ることができるよう、]をエスケープしなくてよい記法として、「ブロックリテラル」を用意しました。
ブロックリテラルは、'\n[[[[\n' で始まり `\n]]]]\n` で終わります。

### 2.6. 値のない名前付き要素
名前が続けて登場した場合は、値が設定されなかったとみなします。

```
(:name :age 36 :sex Male)
```
これは `{'name': '', 'age': '36', 'sex': 'Male'}` となります。

なお、暗黙的に次要素のみが名前に対応した値となるため、複数要素や空白を含む要素の場合は、リテラルや新たなリストによってこれを囲みます。

```
(:name [Joe Smith] :account (普通預金 12345))
```

### 2.7. Sオブジェクト
ここまでの定義から想像されますが、()でくくられた一連のブロックは、他言語においては、名前が定義されない場合は配列、名前が定義される場合は辞書、としての意味を持ちます。
このように、配列にも辞書にもなれる存在となるため、これらを包含して、S式に類似したオブジェクトという意味で、Sオブジェクトと呼ぶことにします。

配列的・辞書的な要素が混在するケースは辞書のように扱い、暗黙的な辞書キーが設定されることになりますが、これは実装に依存するものとし、記法の仕様には含めないとします。
```
(:name [Joe Smith] :profession detective male 36)
```

は
```
{'name': 'Joe Smith', 'profession': 'detective', 'key001': 'male', 'key002': '36'}
```

のように解釈されます（後ろ２件のキーは実装依存）。

### 2.8. 変数
ドキュメントの中で変数を定義し、繰り返し参照することができます。

#### 2.8.1. 変数のスコープ
変数は `@`  で始まる名前です。

#### 2.8.2. 導字とメタ導字
ここまで、 `:` 、 `@` 、 `#` という記号が名前の先頭に用いられ、それぞれ要素名、ローカル変数、パブリック変数を意味することを説明しましたが、このような名前の先頭に付加する記号のことをSTNでは導字と呼ぶことにします。
後続の仕様にて、他の役割の導字を導入します。
更に、この導字の前に用いることで別の役割を持つ記号があり、これをメタ導字と名付けます。

#### 2.8.3. 変数の定義
変数が後続で使用できるようにするために、変数として宣言する必要があります。
変数を定義するには、変数名の前に `@` というメタ導字を使用します。

```
@@document_title 見積書
@@document_date [2026-01-01]
@#tom_and_jerry_like_relationship [トムとジェリーという漫画に出てくるキャラクターのような関係]
```

以降のドキュメント内でこれを使用することができます。

#### 2.8.4. Sオブジェクトの変数と要素へのアクセス
```
@@joe (:name [Joe Smith] :age 36 :sex Male)
```
という定義を行うことで、以降はSオブジェクトの変数として使用することができます。

このとき、各要素に名前でアクセスするには、.名前という記法を使います。
`@joe.name` であれば `'Joe Smith'` という値を意味します。
また、要素名を括弧で列挙することで複数の要素に一度にアクセスすることが可能です。
`@joe.(name age)` であれば、 `([Joe Smith] 36)`というSオブジェクトを返します。
存在しない名前には空要素で返し、エラーは発生しません。

#### 2.8.5. Sオブジェクトと要素への順序アクセス
先に述べたように、Sオブジェクトは配列でもあり辞書でもあるため、配列の添字的なアクセスも可能とします。
この場合も、名前によるアクセスと同じく、添字をドットで付加します。
但し、添字は1開始になります。これは、計算機の都合ではなく、人間が直感的に「何番目の要素」と意識できるようにするためです。

```
@@joe (:name [Joe Smith] :age 36 :sex Male)
に対しては
```
`@joe.1` は `'Joe Smith'` という文字列を、
`@joe.(1 3)` は `([Joe Smith] Male)` というSオブジェクトを参照します。

#### 2.8.6. Sオブジェクト要素へのパスアクセス
```
@@tom (:hand (:right (:finger (:middle injured ))))
```
という階層の深いSオブジェクトがあったとき、
```
@@tom.hand.right.finger.middle
@@tom.(hand).(right).(finger).(middle)
@@tom.1.1.1.1
@@tom.(1).(1).(1).(1)
```
とすれば、この場合は `injured` という値が得られますが、これを
```
@@tom.[hand.right.finger.middle]
```
または
```
@@tom.[1.1.1.1]
```
のように、階層パスをドットで連結したリテラルで与えることで同じ結果を得られます。


### 2.9. 非構造的要素
STNのドキュメントは基本的にSオブジェクトとみなされますが、Sオブジェクトの構造に含まれない要素として、行コメントとデータブロックを記述できます。

#### 2.9.1. 行コメント
行コメントは、各行の中で//で始まる行末までです。ブロックコメントはありません。

@@joe (:name [Joe Smith] :age 36 :sex Male) // joeの定義

#### 2.9.2. 文末データブロック
ドキュメント中
^====+data====+$
というパターンの行が現れた場合、そこから以降はデータとみなしてドキュメント構造には含めません。
このデータブロック全体は`@_DATA`という変数で線芸されたものとみなし、ドキュメント内で参照可能とします。

##### 2.9.2.1 無名データ
文末データブロックだけが設定された場合、`@_DATA._ALL` としてアクセスできる要素とします。

##### 2.9.2.2 名前付きデータ
文末データブロック内に
^----+ @name$
のパターンのある行が現れた場合は、これをセパレータとみなし、次の行から次のセパレータの前までのデータこのnameを用いて`@_DATA.name`のようにアクセスできます。

このとき、このセパレータの前までのデータは`@_DATA._PRE`という名前でアクセスします。
また、名前が重複した場合は、この名前の要素自体をリストみなして`@_DATA.name.1`のようなアクセスを行うこととします。

### 2.10. 型定義
導字`%`を使うことで、型をユーザ定義することができます。
```
@%Person (:firstname :lastname)
```
という型定義を行うことで、firstname、lastnameという２つの要素をもつことを想定したPersonという型を定義することになります。これを用いて、この型をもった変数を宣言することが可能になります。

```
@Joe %Person(:firstname Joe :lastname Smith)
@Hanako %Person(:firstname Hanako :lastname Yamada)
```

#### 2.10.1 インスタンス生成時の引数名省略
定義された型で新しくインスタンスを作る場合、そのメンバーは要素名で指定子ますが、定義された順序通りの場合は名前を省略することができます。

```
@%Person (:firstname :lastname)
@Hanako %Person(Hanako Yamada)
```
の場合は
```
@Hanako %Person(:firstname Hanako :lastname Yamada)
```
と同じ意味になります。

使用するときに定義時よりも少ない引数が与えられた場合は指定して引数までが有効、多い引数が与えられた場合は無視されます。
```
@Hanako %Person(Hanako) // @Hanako.lastname が空
@Hanako %Person(Hanako Yamada Lawyer) // lawyer は無視
```

#### 2.10.2 メンバー型指定
型定義の各メンバーの既定値は文字列ですが、それ以外のデータ型を指定することができます。
```
@%Person (:name :age % :sex %e(F M))
```
この例では、nameメンバーを文字列で、ageメンバーを整数で、sexメンバーをFかMのいずれかの択一で選ぶということを意味します。
このPerson型では次のような変数宣言が想定されます。
```
@Joe %Person(:name [Joe Smith] :age 36 :sex $e(M))
@Hanako %Person(:name [Hanako Yamada] :age 32 :sex $e(F))
```

#### 2.10.3 選択要素
先の例に出てきた例で
```
%e(F M)
```
のような形で、選択要素として想定されているメンバーに値を与えるとき、文字列で設定すればその文字列を意味する選択値が適用されたものとみなされます。
つまり、
```
@Joe %Person(:name [Joe Smith] :age 36 :sex M)
```
は、
```
@Joe %Person(:name [Joe Smith] :age 36 :sex $e(M))
```
と同じ意味になります。


#### 2.10.4 使用できる型名

メンバー型定義に使用する型名は以下のとおりです。

| 型名 | 意味 |
|----|----|
| （指定なし） | 文字列 |
| % / %% / %i / %int | 整数 |
| %f / %fl / %float | 浮動小数点 |
| %$ / %cur / %currency | 通貨 |
| %d / %dt / %date | 日付 |
| %t / %time / %datetime | 日付時刻 |
| %b / %bln / %boolean | 論理値 |
| %e / %en / %enum | %e(...) の形式で選択肢 |
| %() | Sオブジェクト（） |

このほか、既に宣言したユーザ定義型をメンバー型指定に用いることもできます。

```
@%Person (:name :age % :sex %e(F M))
@%Task (:by %Person :taskname :deadline %d)
```

#### 2.10.5 要素値の後設定
先の例では、型を用いた変数をインスタンス化するときに要素値を指定しましたが、既にインスタンス化された変数に対して後から要素値を設定することもできます。
このためには、`!`導字をに要素名を後置し、その引数に設定値を指定します。

```
@%Person (:name :age % :sex %e(F M))
@@joe %Person(:name joe :age 36)!sex(%e(M))
```
先に見たドットアクセスは、要素値の取得つまりgetterでしたが、`!`はsetterであると言えます。
setterは戻り値としてその変数そのもを返すため、メソッドチェーンが可能です。

```
@@joe %Person(:name joe)!(age 36)!sex(%e(M))
```

##### 2.10.5.1 後設定は上書き
後設定した要素値は、後で指定した値によって上書きされます。

##### 2.10.5.2 宣言のない要素の指定
型指定で宣言していない要素についても、後付けで要素追加が可能です。
```
@@joe %Person(:name joe)!age(36)!sex(%e(M))!desc([This is a man named Joe.])
```
この後付要素追加は、スキーマ定義を柔軟に行うことを可能にし、必ずしも常に存在するわけではない要素のインスタンス固有の追加や、オブジェクト本体の性質の記述ではない注釈的な要素を追加することができるようになります。
こうしたSparceな要素の定義を行うスキーマ言語は珍しい存在といえます。

##### 2.10.5.3 要素の一括指定
メソッドチェーンしていく他に、要素を一括して設定することも可能です。
このためには、``!+(:name value ...)``の形式で指定します。
こちらも設定したオブジェクト自体を返すのでその後のメソッドチェーンも可能です。

```
@@joe %Person(:name joe)
@joe!+(:age 36 :sex %e(M) :desc [This is a man named Joe.])
```
この形式はインスタンス時の要素指定と結果としては同じですが、設定タイミングを分割することが可能になります。


#### 2.10.6 要素値の出現回数
#### 2.10.6.1 コレクション
要素値の先頭に`*`を付加することで、その要素が複数回出現することを表します。

```
@%Task (:name :duedate %d)
@%Person (:name :tasks *%Task)
@@joe %Person(:name joe :tasks 
  (
    %Task (:name 牛乳を買う :duedate [02-16])
    %Task (:name 家賃を払う :duedate [02-20])
  )
)
```

#### 2.10.6.2 選択型と拡張
2.10.2 で`%e`を選択型として挙げていましたが、これによって特定の集合からの択一を表現できます
```
@%Person (:name :sex %e(F M))
@ken (:name [Ken Ito] :sex %e(M))
//または @ken (:name [Ken Ito] :sex M)
@ami (:name [Ami Sato] :sex %e(F))
// または @ami (:name [Ami Sato] :sex F)

```

この前に `*` を付加することで、複数選択型を表現することができます
```
@%Musician (:name :parts *%e(Vo EG AG Ds b Key))
@paul (:name [Paul McCarteney] :parts %e(Vo b))
@john (:name [John Lennon] :parts %e(Vo EG))
```

#### 2.10.7 無名要素
独立した型として再利用する必要がなく、名前をつける必要もない場合は、無名要素として定義することが来ます。
無名要素は`%(...)`として、ネストしたカッコ内に更に型定義を含めることで実現します。

```
@%Person (:name :tasks *%(:lab :dudate %d))
@@joe %Person(:name joe :tasks 
  (
    %(:lab 牛乳を買う :duedate [02-16])
    %(:lab 家賃を払う :duedate [02-20])
  )
)
```



#### 2.11 型拡張（ポリモーフィック）
複数の型に共通の要素を見つけ、ポリモーフィックに処理したいような場合は、パターンマッチを使った型の拡張を行うことができます。

```
@%Content ~(
  ( :type %e(text file) )
  (
    :(:type text) (:text)!as(%TextContent)
    :(:type file) (:fileDiscriptor :fileSize)!as(%FileContent)
  )
)
@@text1 %TextContent(:text これはテキストです)
@@text2 %TextContent(:text [This is a text.])
@@File1 %FileContent(:fileDiscriptor [file1.pdf] :fileSize [406k])
@@File2 %FileContent(:fileDiscriptor [file1.jpg] :fileSize [516])
```
ここで、 `~()` はパターンマッチを行うための構文です。
パターンマッチは、 `~((＜条件の変数＞) (＜パターンの列挙＞))` として、各パターンを
 `~(:(＜条件＞) (＜その条件の場合の値＞)` という、辞書エントリーのような形で表現します。
つまり、この構文では、%Contentという型が、typeという種類に分かれて、typeの値ごとに別の派生型を生成する、という意味となります。
この構文のときに、各パターンに`!as(%型名)`という修飾子を用いることで、型名として宣言する仕様になります。

#### 2.12 予約要素
オブジェクト内に指定するメンバー要素の内、特殊な意味を持つ要素を「予約要素」として別管理にできます。
予約要素は、`__`（アンダーバー2つ）で表される要素であり、それ自体の内部にまた名前付き要素を持ちます。
これは型・インスタンスの両方に有効です。

```
@%Organization (:__ (:type Organization))
```

#### 2.12.1 予約要素へのアクセス
予約要素も通常の要素と同じくドットまたはブロック／リテラルによるアクセスが可能です
```
@%Organization (:__ (:type Organization))
@%Organization.__.type // -> Organization
@%Organization.[__.type] // -> Organization
```
但し、順序アクセスの対象にはなりません。

#### 2.12.2 予約要素への値の設定
予約要素は、型によるパターンマッチや、インスタンスのid等に使用されることが想定されるため、一度だけ設定したら上書きすることはできません。

```
@%Organization (:__ (:type Organization)) //->変更不可
```
```
@%Person (:name) // 型定義
@%Person!__(:type Person)) //->定義後に一度だけ設定可
```

#### 2.12.3 順序引数による初期化は非対象

「2.10.1 インスタンス生成時の引数名省略」で見た通り、定義時の順通りであれば引数名は省略できるとしていますが、予約要素はこれに含みません。

```
@%Person (:__ (:type Person) :firstname :lastname)
%Person(Hanako Yamada) // __ではなく、firstname=Hanako, lastname=Yamadaが入る
```

#### 2.12.4 予約要素__(:id)のショートカット
予約要素のうち、`id`という名前の要素については、予約要素を通さないショートカットとして使用できます。

なお、直接に`id`という名前の要素を追加することは問題ありませんが、特別な意味を持ちません。

```
@@members (
  %Person (:__(:id taro) :name Taro :age 30)
  %Person (:__(:id jiro) :name Jiro :age 29)
)
```
のように定義した`@members`変数で
```
@members.(taro)
```
のようにすることで`%Person(:name Taro)`を得ることができます。
この形は既に説明したとおりですが、この設定の短縮形として以下の操作を可能とします。

```
@@members (
  %Person (:name Taro :age 30)!(taro)
  %Person (:name Jiro :age 29)!(jiro)
)
```
つまり、後続の名前が何もつかない単独の`!`セッターは、予約要素の__(:id)を付与する意味を持ち、

```
@members.(taro)
```
と、同じ形の`.`ゲッターで要素を取得できます。

#### 2.13 パック展開
```
@@taro %Person(:name [Taro Yamada] :age 36 :sex M)
```

このとき、@taro には %Person のインスタンスが設定されます。
同時に、インスタンス生成に用いた要素の値（:name / :age / :sex）が存在します。

これを分解して複数の変数に同時に設定したい場合、左辺に変数のリストを与えます。

```
@(@taroname @taroage @tarosex) %Person(:name [Taro Yamada] :age 36 :sex M)
```

このように定義すると、右辺で指定された値が順に分解され、

@taroname に :name の値
@taroage に :age の値
@tarosex に :sex の値

がそれぞれ設定されます。これをパック展開とします。

#### 2.13.1 要素によるパック展開
前項は右辺がリスト表現になっている場合で、直接左辺にマップされましたが、右辺の特定の要素にリスト表現が含まれる場合、`**`（アスタリスク2個）を使うことによって、そのリストを順次適用したパック展開をすることができます。

```
@(@taro @jiro @hanako) %Person(:name **(Taro Jiro Hanako))
```
型・予約要素にも同様のことが可能です。
```
@(%Organization %Person %Event) (:__ (:type **(Organization Person Event)))
```

#### 2.14 型の宣言と初期化の一括処理
型宣言のブロックと初期化のブロックを続けて書くことによって、一括で指定できます。
```
@@taro (%Person(:name :age % :sex %e(F M)) ([Taro Ito] 26 M))
```
この場合、この宣言ブロック内においては `%Person` 型を何度も使用できることになります。
一方、その型を名前として何度も使う必要がなければ、無名型として
```
@@taro (%(:name :age % :sex %e(F M)) ([Taro Ito] 26 M))
```
という一時宣言であれば、新しく `%Person` 型を定義することなく初期化可能です。


#### 2.15 シンボル
導字`#`で始まる名前は「シンボル」というワントークンとして認識されます。

シンボルは変数に似ていますが、値の再利用というより、ドキュメント内の「キー」として柔軟に使用できます。

変数と同様の宣言を行う場合

```
@#taro (%Person(:name Taro))
```

オブジェクト発生時に登録する場合
```
Person(:name Taro)!#(#taro)
```

（2.12.4 予約要素idで表した単独のセッターとは異なり、#1文字のセッター、と考えることができます。


#### 2.16 ロケータ
ドキュメントの様々な要素にアクセスするための「ロケータ」についてまとめます。

##### 2.16.1 オブジェクト内ロケータ
SObjectの内部については、既出の通り、`.` によるアクセスが可能です。
これらは順序インデックスと、要素名としての指定、そして予約要素__(:id)による要素名の定義がある場合に有効です。
（2.8.4、2.8.5、2.12.4）
```
//順序インデックス
@@members (
  %Person (:name Taro)
)
@members.(1)
// 要素名
@@members (
  :taro %Person (:name Taro)
)
@members.(taro)
// 要素名にシンボルを使用
@@members (
  :#taro %Person (:name Taro)
)
@members.(#taro)
// 要素要素__(:id)の使用
@@members (
  %Person (:__(:id #taro) :name Taro)
)
@members.(#taro)
// 予約要素__(:id)後置
@@members (
  %Person (:name Taro)!(#taro)
)
@members.(#taro)
```
なお、名前と予約要素__(:id)の両方を指定した場合は両方のロケータが有効になります。
```
// 要素名の指定と予約要素__(:id)の後置
@@members (
  :taro %Person (:name Taro)!(#taro)
)
@members.(taro)
@members.(#taro)
```

##### 2.16.2 オブジェクト内クエリーロケータ

更に、インデックス/名前/予約要素以外のロケータとして、`?` によるクエリーを行うことができます。

```
@@members (
  %Person (:name Taro :age 30 :sex M)!(#taro)
  %Person (:name Hanako :age 31 :sex F)!(#hanako)
  %Person (:name Tom :age 29 :sex M)!(#tom)
  %Person (:name Jane :age 28 :sex F)!(#jane)
)
@members?(:name Tom) //=> %Person (:name Tom :age 29 :sex M)!(#tom)
@members?(:sex F) //=> (%Person (:name Hanako ... ) %Person (:name Jane)  ... )

```

##### 2.16.3 ドキュメント内クエリーロケータ

2.15で、オブジェクトにシンボルを設定した場合は、`#`をロケータに用いることができます。

```
// 次のいずれの場合も
@#taro %Person(:name Taro)
%Person(:name Taro)!#(#taro)

// このようにロケートされる
#(#Taro)
```
